## 1. 컴포넌트 레이어 규칙

모든 컴포넌트는 4개 레이어 중 하나에 속하며, **레이어 간 사용 방향은 단방향**이다.

```
Atom  →  Organism  →  화면(Screen)
                  ↑
            Utility (디자이너 전용, 화면 배포 불가)
```

| 레이어 | 예시 | 화면 직접 배치 |
|--------|------|--------------|
| Atom | Button, Ico, Badge, RadioText | ✗ 불가 |
| Single Organism | AppBar, TextField, Chip, ListText | ✓ 가능 |
| Compound Organism | CardSection, Bottomsheet, Popup, CompletePage | ✓ 가능 |
| Utility | Spacing, TextFieldDefault 상태 프레임 | ✗ 불가 |

**핵심 원칙**: Button을 화면에 직접 올리지 않는다. 반드시 CardSection의 CTA 슬롯, Sticky CTA 영역, Popup의 ActionButton 슬롯 등 **Organism이 제공하는 슬롯** 안에서만 사용한다.

## 2. 크롬 조합 패턴

화면 유형에 따라 상단·하단 고정 영역(크롬)의 조합이 달라진다.

### 2.1 화면 유형별 크롬 매트릭스

| 화면 유형 | StatusBar | AppBar variant | 상단 탭 | 하단 영역 |
|----------|-----------|---------------|---------|---------|
| **메인 탭** | 기본 | 홈 전용 (T로고) | UnderlineTab (관리/검색/쇼핑) | BottomNavigation |
| **리스트** | 기본 | 서브 기본 (←) | UnderlineTab (선택) + Chip 필터 (선택) | 없음 |
| **상세_상품** | 투명 overlay | 투명 (bg=transparent) | 없음 | Sticky CTA |
| **상세_폼** | 기본 | 서브 기본 (←) | 없음 | Sticky CTA |
| **완료** | 다크 (아이콘 반전) | 없음 (X 버튼만) | 없음 | Sticky 2-버튼 |
| **바텀시트** | — | — | UnderlineTab (필터형만) | CTA 영역 (선택) |
| **팝업** | — | — | 없음 | PopupActionButton |

### 2.2 AppBar variant 선택 기준

```
홈 전용   : 메인 탭(MY·검색·쇼핑) 화면에서만 → T로고 + 스캔/장바구니/메뉴
서브 기본  : 하위 depth 진입 시 → ← 뒤로가기 + 타이틀 + 선택 아이콘
서브 검색  : 리스트 화면에서 검색 기능 필요 시 → ← 뒤로가기 + 타이틀 + 검색
투명      : 상세 상품 화면 → Hero 이미지와 겹치는 경우
```

### 2.3 BottomNavigation vs Sticky CTA 선택

```
BottomNavigation  : 메인 탭 화면(깊이 0)에서만. 탭 전환이 주 목적
Sticky CTA        : 서브 화면 전체. 단계 진행 또는 상품 구매가 주 목적
둘 다 없음        : 완료 페이지(2-버튼 있지만 Sticky CTA와 다른 컴포넌트)
                    바텀시트/팝업(자체 CTA 슬롯 사용)
```

### 2.4 상단 탭(UnderlineTab) 추가 조건

UnderlineTab은 두 곳에서만 등장한다.

1. **메인 탭 화면** → 항상 포함 (관리/검색/쇼핑 3탭 고정)
2. **리스트 화면** → 콘텐츠를 2개 이상의 카테고리로 분류할 때
3. **바텀시트** → 필터형 시트에서만 (다항목 필터 선택 시)

UnderlineTab 아래에 **Chip 필터 행**을 추가할 수 있다. 단, 탭이 대분류라면 Chip은 소분류 역할을 해야 하며 둘이 같은 축의 분류를 중복으로 표현하지 않는다.

## 3. 페이지 조합 패턴

### 3.1 메인 탭 (MY 관리)

```
StatusBar
AppBar [홈 전용]
────────────────────────────────  ← 상단 고정 크롬 끝
UnderlineTab [관리/검색/쇼핑]
════════════════════════════════  ← 스크롤 시작
BannerBenefit              ← 멤버십 혜택 배너. 항상 스크롤 최상단
CardSection × N            ← 정보 카드 반복 (결합가족, 잔여량, 요금안내서 등)
LocalButtonSection         ← 빠른 실행 2분할 메뉴 (T가족모아/데이터함께쓰기)
BannerHorizontalMedium     ← 프로모션 배너. 카드 사이에 1개만
CardSection × N            ← 추가 정보 카드 (약정할인, T멤버십 등)
ButtonItem [MY 편집]       ← 스크롤 최하단. 카드 편집 진입 트리거
════════════════════════════════  ← 스크롤 끝
BottomNavigation           ← 하단 고정
```

**조합 규칙**:
- CardSection은 같은 성격의 정보끼리 묶는다. 결합가족·잔여량·요금안내서는 각각 별도 카드.
- BannerHorizontalMedium은 CardSection 그룹 사이에 1개 삽입. 2개 연속 금지.
- LocalButtonSection(빠른실행)은 항상 카드 그룹 사이에 배치. 최상단/최하단 배치 금지.

### 3.2 메인 탭 (쇼핑)

```
StatusBar
AppBar [홈 전용]
UnderlineTab [관리/검색/쇼핑]
════════════════════════════════
Callout [프로모션 배너]          ← full-width, 스크롤 최상단
──────────────────────────────
TitleSection + CarouselProduct  ← 카테고리 1
TitleSection + CarouselProduct  ← 카테고리 2  (패턴 반복)
──────────────────────────────
TitleSection + ThumbnailItem 2열 그리드
──────────────────────────────
Footer
════════════════════════════════
BottomNavigation
```

**조합 규칙**:
- 각 상품 카테고리 섹션은 반드시 `TitleSection + 콘텐츠` 쌍으로 구성.
- 썸네일 이미지 중심 상품 → CarouselProduct(횡스크롤) 또는 ThumbnailItem 2열 그리드.
- Footer는 쇼핑 탭에서만 노출. 다른 탭에는 없음.

### 3.3 리스트_카드

```
StatusBar
AppBar [서브 기본]
Chip 필터 행 (선택)            ← 카테고리 필터가 필요할 때
FilterSorting (선택)           ← 결과 수 표시 + 정렬이 필요할 때
════════════════════════════════
ThumbnailItem 2열 그리드       ← 이미지 중심 상품 (구독상품, 혜택)
또는
ListItem 1열                   ← 텍스트 정보 중심 상품 (요금제, 단말기)
════════════════════════════════
```

**레이아웃 선택 기준**:

| 콘텐츠 | 레이아웃 | 이유 |
|--------|---------|------|
| 구독상품, 혜택 | 2열 ThumbnailItem | 이미지로 식별 가능, 비교보다 탐색 목적 |
| 요금제, 부가서비스 | 1열 ListItem | 텍스트 정보량이 많아 2열 시 잘림 |
| 단말기 | 1열 ListItem | 가격+스펙 텍스트 필수 표시 |
| 인터넷 상품 | 1열 카드 | 상품 수 적고 각 상품 정보가 상세 |

### 3.4 리스트_텍스트

```
StatusBar
AppBar [서브 기본]
UnderlineTab (선택)            ← 카테고리 분류 필요 시 (이용내역 유형 등)
Summary Header (선택)          ← 총합 수치가 있을 때 (포인트 잔액 등)
Chip 기간 필터 (선택)          ← 기간/유형 필터 필요 시
════════════════════════════════
날짜 그룹 헤더                 ← 날짜 기준 그루핑이 있는 경우
ListText × N                   ← 거래/내역 반복 아이템
날짜 그룹 헤더
ListText × N
════════════════════════════════
```

**조합 규칙**:
- Summary Header는 UnderlineTab 아래에 위치. 탭 필터와 겹치지 않도록 탭 선택과 Summary가 연동되어야 함.
- 날짜 그룹 헤더는 ListText 위에만 붙는다. Divider로 대체하지 않는다.

### 3.5 상세_상품

```
StatusBar [투명]
AppBar [투명]               ← Hero 이미지 위에 float
════════════════════════════════
Hero 이미지                 ← full-width. StatusBar/AppBar 아래에서 시작 (overlap)
상품명 + Badge + 별점
가격 블록                   ← 원가(취소선) + 할인가 + 월할부 세트로 묶음
──────────────────────────────
TitleSection + 주요 혜택 목록
──────────────────────────────
AccordionProductInfo × N    ← 상품 상세, 이용안내, 주의사항 등
──────────────────────────────
TitleSection + CarouselProduct   ← 관련 상품 추천 (필수)
Footer
════════════════════════════════
Sticky CTA                  ← 구매/신청 버튼 하단 고정
```

**조합 규칙**:
- Hero 이미지와 투명 AppBar는 항상 세트. 투명 AppBar 없이 Hero 단독 사용 금지.
- 관련 상품 CarouselProduct는 상세 페이지의 필수 요소. Footer 위에 반드시 포함.
- Accordion은 항상 `TitleSection 없이` 단독 나열. 각 항목이 자체 타이틀을 가짐.

### 3.6 상세_폼 (정보입력 플로우)

폼 화면은 4단계 플로우로 구성되며, 각 단계는 독립된 화면이다.

```
[인풋] ─────────────────────────────────────
  TitleSection (단계 안내)
  TextField 그룹 × N
  CheckboxText 목록 (약관)
  Sticky CTA → "다음"

[정보체크] ──────────────────────────────────
  InfoTextList × 섹션     ← 입력 정보 확인 (읽기 전용)
  RadioText 목록 (선택 옵션 있을 때)
  Sticky CTA → "확인"

[장바구니] ──────────────────────────────────
  상품 카드 × N
  AccordionPriceInfo      ← 가격 구성 펼침
  InfoTextList [합계]     ← 최종 금액 요약
  Sticky CTA → "주문하기"

[결제] ──────────────────────────────────────
  주문 요약 카드
  ListSelected (결제수단 선택)
  InfoTextList (가격 상세)
  CheckboxText (약관 전체 동의)
  AccordionPriceInfo × N  ← 각 약관 내용
  Sticky CTA → "약관에 동의하고 결제하기"
```

**조합 규칙**:
- 각 단계의 AppBar 타이틀은 현재 단계명을 표시. (예: "결제하기")
- 가격 관련 정보는 반드시 `AccordionPriceInfo(상세) + InfoTextList(합계)` 쌍으로 구성.
- 최종 결제 단계의 CheckboxText는 약관 전체 동의가 목적이므로 AccordionPriceInfo와 연결되어야 함.

### 3.7 완료

```
전체 배경 [dark gradient]
StatusBar [아이콘 색 반전]
X 닫기 버튼 (AppBar 대체)
──────────────────────────────
완료 제목 + 서브텍스트        ← 친근한 구어체 ("OO되었어요")
──────────────────────────────
InfoTextList 카드             ← 처리 결과 요약 (반투명 카드)
──────────────────────────────
[결제 완료만] CarouselProduct ← 크로스셀 (Footer 역할)
════════════════════════════════
Sticky 2-버튼 [Secondary + Primary]
```

**조합 규칙**:
- AppBar를 사용하지 않는 유일한 화면. X 버튼이 AppBar를 대체.
- InfoTextList는 반드시 반투명 카드(별도 컨테이너) 안에 담긴다. 배경에 직접 올리지 않음.
- 완료_결제만 CarouselProduct를 포함. 다른 완료 화면(개통/요금제변경/해지)은 포함 않음.
- 하단 버튼은 항상 `Secondary(추가 액션) + Primary(확인/홈으로)` 구조.

## 4. 카드 조합 패턴

### 4.1 CardSection 내부 구조

CardSection은 `타이틀 영역 + 콘텐츠 영역`의 2-슬롯 구조다.

```
CardSection
  ├── 타이틀 슬롯
  │     ├── 서브 레이블 (카드 용도 명시)
  │     ├── 주요 수치 (핵심 정보)
  │     └── BadgeHome (보조 맥락, 선택)
  │
  └── 콘텐츠 슬롯 (선택)
        ├── 텍스트/수치 상세
        ├── 바코드 또는 이미지
        ├── AI 추천 텍스트
        └── 내부 CTA 버튼 (1개 제한)
```

**카드 타입별 구성**:

| 카드 용도 | 타이틀 슬롯 | 콘텐츠 슬롯 | 우측 아이콘 |
|----------|-----------|-----------|-----------|
| 단순 수치 표시 | 서브레이블 + 수치 + Badge | 없음 | 관련 아이콘 40px |
| AI 진단/추천 | 서브레이블 + 멀티라인 텍스트 | AI 추천 텍스트 | 없음 |
| 추천 + CTA | 서브레이블 + 제목 텍스트 | 내부 CTA 버튼 | 없음 |
| 바코드 | 서브레이블 | 바코드 이미지 + 번호 | 없음 |
| 빠른 실행 | — | 2분할 메뉴 (아이콘+텍스트) | — |

**조합 규칙**:
- 카드 내부 CTA 버튼은 최대 1개. 여러 액션이 필요하면 Sticky CTA 영역으로 이동.
- 우측 아이콘(40px)은 수치 표시 카드에만 사용. AI/추천 카드에는 없음.
- BadgeHome(회색 pill 배지)은 수치의 맥락을 보완할 때만 사용. (예: "80GB" 중 "32GB" 표시 시 "80GB" 배지)

### 4.2 상품 카드 (ThumbnailItem) 내부 구조

```
ThumbnailItem
  ├── 썸네일 이미지 (상단)
  ├── 상품명 (중간)
  ├── 가격 블록
  │     ├── 원가 (취소선)
  │     └── 할인가 (#3617CE)
  └── CTA 버튼 (선택)
```

**조합 규칙**:
- 원가와 할인가는 항상 세트. 할인이 없어도 원가만 표시하거나 할인가만 표시하지 않음.
- 2열 그리드 배치 시 모든 카드의 높이가 동일해야 함 (CTA 유무 통일 필요).

## 5. 리스트 조합 패턴

### 5.1 리스트 컴포넌트 선택 기준

| 상황 | 사용 컴포넌트 | 이유 |
|------|------------|------|
| 거래/활동 내역 (날짜 + 금액) | `ListText` | 좌측 텍스트 + 우측 값 구조 |
| key-value 정보 나열 (확인·결제 화면) | `InfoTextList` | 항목명-값 정렬이 목적 |
| 선택 가능한 옵션 목록 | `ListSelected` | 라디오/체크 + 텍스트 구조 |
| 바텀시트 내 단순 선택 | `RadioText` / `CheckboxText` | 바텀시트 슬롯에 최적화 |
| FAQ/안내 아코디언 | `AccordionNoticeInfo` | Q&A 형식 |
| 상품 상세/스펙 아코디언 | `AccordionProductInfo` | 상품 정보 펼침 |
| 결제 금액 구성 아코디언 | `AccordionPriceInfo` | 금액 항목 펼침 |

### 5.2 날짜 그룹핑이 필요한 경우

`날짜 그룹 헤더 + ListText 묶음` 패턴은 이용내역·포인트내역·할인내역에서 사용한다.

```
날짜 그룹 헤더 "2026.03"    ← full-width 배경 블록
  ListText (거래 1)
  ListText (거래 2)
날짜 그룹 헤더 "2026.02"
  ListText (거래 3)
```

Divider로 날짜를 구분하지 않는다. 날짜 그룹 헤더 컴포넌트를 반드시 사용한다.

### 5.3 금액 합계가 있는 InfoTextList 구성

결제·정보체크 화면의 금액 요약은 항상 아래 구조를 따른다.

```
InfoTextList
  일반 행    : 항목명 — 금액
  일반 행    : 항목명 — 금액
  할인 행    : 항목명 — 금액 (빨간색, - 표시)
  ─────────────────────────── Divider H  ← 반드시 포함
  합계 행    : 최종 결제 금액(굵게) — 금액(파란색, 굵게)
```

합계 행 위에 Divider H가 없으면 합계임을 시각적으로 구분할 수 없으므로 필수다.

## 6. 오버레이 조합 패턴

### 6.1 바텀시트 vs 팝업 선택 기준

| 상황 | 선택 | 이유 |
|------|------|------|
| 옵션 목록에서 하나 선택 | 바텀시트 | 스크롤 가능한 목록 수용 |
| 여러 조건 필터 설정 | 바텀시트 (탭 포함) | 필터 항목이 많아 스크롤 필요 |
| 약관 동의 항목 나열 | 바텀시트 | 체크 항목 수가 많을 수 있음 |
| 상품 간략 정보 + 구매 확인 | 바텀시트 | 정보와 CTA를 함께 표시 |
| 단순 확인/취소 (2줄 이내) | 팝업 | 콘텐츠가 짧아 바텀시트 과도함 |
| 결제 실패·에러 알림 | 팝업 | 즉각 인지 필요, 짧은 메시지 |
| 개인정보 동의 | 팝업 | 체크박스 1–2개, 짧은 동의 절차 |

**판단 기준**: 콘텐츠가 스크롤되거나 목록이 3개 이상이면 바텀시트. 메시지 + 버튼 구조면 팝업.

### 6.2 바텀시트 내부 조합 패턴

바텀시트는 항상 `핸들 + 타이틀 + 콘텐츠 슬롯 + [CTA]` 구조다.

```
바텀시트 유형별 콘텐츠 슬롯 구성:

[단순 선택형]
  핸들
  타이틀 (17px/600) + X 닫기
  ─────────────────────────────
  RadioText × N               ← 회선 선택, 배송방법 등
  ─────────────────────────────
  CTA [Primary 1개]

[다항목 필터형]
  핸들
  타이틀 + X 닫기
  UnderlineTab               ← 필터 카테고리 탭
  ─────────────────────────────
  RadioText × N              ← 탭별 필터 항목
  ─────────────────────────────
  CTA ["N개 상품 보기"]

[약관 동의형]
  핸들
  타이틀 + X 닫기
  ─────────────────────────────
  CheckboxText [전체 동의]    ← 전체 동의 항목 항상 최상단
  Divider H
  CheckboxText [필수] × N
  CheckboxText [선택] × N
  ─────────────────────────────
  CTA ["동의하기"]

[상품 요약형]
  핸들
  타이틀 + X 닫기
  ─────────────────────────────
  상품 이미지 + 상품명 + 가격 블록
  InfoTextList (옵션 상세)
  ─────────────────────────────
  CTA [Secondary "닫기" + Primary "구매하기"]

[상품 추천형]
  핸들
  타이틀 + X 닫기
  ─────────────────────────────
  안내 텍스트
  CarouselProduct
  ─────────────────────────────
  CTA 없음 (X 버튼만으로 닫기)
```

**조합 규칙**:
- 바텀시트 안에 또 다른 바텀시트를 중첩하지 않는다.
- UnderlineTab은 필터형 바텀시트에서만 사용. 단순 선택형에는 없음.
- 약관 동의형에서 "전체 동의" 체크박스는 반드시 최상단에 위치하며, Divider로 개별 항목과 분리한다.

### 6.3 팝업 내부 조합 패턴

팝업은 항상 `타이틀 + 본문 + 버튼` 구조다.

```
[동의형]
  타이틀
  본문 텍스트 (짧은 설명 + 링크)
  CheckboxText [필수/선택 동의]
  ─────────────────────────────
  PopupActionButton [다음에 + 동의하고 시작]

[정보형]
  타이틀
  InfoTextList (주요 정보)
  ─────────────────────────────
  PopupActionButton [닫기 + 추가 액션]

[에러형]
  타이틀
  본문 텍스트 (에러 설명)
  ─────────────────────────────
  PopupActionButton [확인 1개, full-width]
```

**조합 규칙**:
- 팝업 내에 스크롤이 생기는 구성 금지. 콘텐츠가 길면 바텀시트로 변경.
- 버튼은 반드시 `.PopupActionButton` 컴포넌트 사용. 일반 `.Button`을 직접 배치하지 않음.

## 7. CTA 조합 패턴

### 7.1 CTA 위치 결정

| 화면 상황 | CTA 위치 | 이유 |
|----------|---------|------|
| 단일 페이지 액션 | Sticky CTA (화면 하단 고정) | 항상 접근 가능해야 함 |
| 카드 내 부가 액션 | 카드 내부 버튼 | 해당 카드에만 종속된 액션 |
| 섹션 더보기 | TitleSection 우측 텍스트 링크 | 탐색 목적, CTA 강도 낮음 |
| 오버레이 확인 | 바텀시트/팝업 자체 CTA | 오버레이 문맥에 종속 |

**CTA를 콘텐츠 중간에 배치하지 않는다.** Primary 버튼은 반드시 Sticky 영역 또는 Organism의 CTA 슬롯 안에 위치한다.

### 7.2 버튼 개수 및 조합 규칙

```
단일 Primary      : 단계 진행, 단순 확인, 취소 불필요한 구매
                    예: "다음", "구매하기", "확인"

Secondary + Primary: 취소 가능 또는 대안 제공 시
                    비율: Secondary 1/3 + Primary 2/3  (주 액션 강조)
                    또는  Secondary 1/2 + Primary 1/2  (동등한 선택지)
                    예: "닫기 + 구매하기", "이전 + 다음"

텍스트 링크만      : 탐색 목적 (더보기, 전체보기)
                    TitleSection 우측 또는 섹션 하단
```

**버튼 타입 선택**:

| 액션 성격 | 버튼 타입 |
|----------|---------|
| 주 전환 액션 (구매, 신청, 결제) | Primary |
| 취소, 이전, 닫기 | Secondary |
| 보조 탐색, 더보기 | Ghost 또는 텍스트 링크 |
| 비활성화 조건 있음 | Disabled (활성 조건 충족 전) |

### 7.3 완료 화면 2-버튼 패턴

완료 화면의 하단 버튼은 항상 `Secondary + Primary` 쌍이며, 각 버튼의 역할이 정해져 있다.

| 버튼 위치 | 역할 | 텍스트 예시 |
|---------|------|-----------|
| 좌측 Secondary | 현재 완료와 관련된 추가 탐색 | "데이터 돌리기", "요금제 보기", "다른 구독 알아보기" |
| 우측 Primary | 플로우 종료, 홈 복귀 | "확인", "홈으로" |

## 8. 폼 조합 패턴

### 8.1 TextField 그룹 구성

관련 있는 입력 필드는 그룹으로 묶고, 그룹 간에는 명확한 제목과 여백으로 구분한다.

```
[그룹 제목]                  ← 15px/600, 그룹 시작 상단에 배치
  TextField [이름]
  TextField [생년월일]
  TextField + 보조 버튼       ← 인증번호 입력처럼 버튼 필요 시
                               TextField flex-grow + Button 고정폭

[그룹 제목]
  TextField [주소]
  TextField [상세 주소]
```

**조합 규칙**:
- 그룹 제목 없이 TextField만 나열하지 않는다. 반드시 그룹 제목이 선행한다.
- TextField + 버튼 조합 시, 버튼은 TextField 내부 우측에 배치. TextField 바깥에 버튼을 나란히 두지 않는다.

### 8.2 에러 상태 조합

```
TextField [Error variant]
HelpText [에러 메시지]        ← TextField 컴포넌트 내 하단 슬롯 사용
```

에러 메시지는 반드시 해당 TextField 바로 아래에 붙는다. 에러 상태인 필드 위에 별도 Callout로 에러를 표시하지 않는다.

### 8.3 약관 동의 구성

약관 동의가 필요한 화면에서의 패턴:

```
인풋 화면 (일반 약관):
  CheckboxText [전체 동의]    ← 항상 최상단
  ─ Divider H ─
  CheckboxText [필수] 약관1
  CheckboxText [필수] 약관2
  CheckboxText [선택] 약관3

결제 화면 (결제 관련 약관):
  CheckboxText [전체 동의]
  AccordionPriceInfo × N      ← 각 약관 내용 펼침 (체크박스와 연결)
```

**조합 규칙**:
- 전체 동의 CheckboxText는 항상 목록 최상단. Divider로 개별 항목과 분리.
- 결제 화면에서는 CheckboxText와 AccordionPriceInfo가 1:1로 연결된다. 약관 내용을 볼 수 있는 Accordion이 없는 CheckboxText 단독 사용 금지.

## 9. 금지 조합

### 크롬 금지

| 금지 패턴 | 올바른 대안 |
|----------|-----------|
| 서브 화면에 BottomNavigation 포함 | Sticky CTA 사용 |
| 완료 화면에 AppBar (← 뒤로가기) 포함 | X 닫기 버튼만 사용 |
| 메인 탭 AppBar에 뒤로가기(←) 포함 | 홈 전용 variant 사용 |
| BottomNavigation + Sticky CTA 동시 사용 | 하나만 선택 |

### 오버레이 금지

| 금지 패턴 | 올바른 대안 |
|----------|-----------|
| 바텀시트 안에 또 다른 바텀시트 | 단계를 새 화면으로 분리 |
| 팝업 내부에 스크롤 | 바텀시트로 교체 |
| 팝업에 일반 .Button 직접 배치 | .PopupActionButton 사용 |

### 컴포넌트 조합 금지

| 금지 패턴 | 올바른 대안 |
|----------|-----------|
| Atom(.Button 등)을 화면에 직접 배치 | Organism의 CTA 슬롯 사용 |
| CTA Primary 버튼을 콘텐츠 중간에 배치 | Sticky CTA 또는 카드 내부 슬롯 |
| 날짜 그루핑을 Divider로 처리 | 날짜 그룹 헤더 컴포넌트 사용 |
| 금액 합계 행 위에 Divider H 누락 | 합계 행 상단 Divider H 필수 |
| 상세 페이지에서 관련 상품 CarouselProduct 누락 | 필수 포함 (Footer 위) |
| 투명 AppBar 없이 Hero 이미지만 단독 사용 | 투명 AppBar와 항상 세트 |

### 콘텐츠 금지

| 금지 패턴 | 올바른 대안 |
|----------|-----------|
| Primary 버튼에 명사형 레이블 | 동사형 사용 ("구매" → "구매하기") |
| Secondary 버튼에 동사형 레이블 | 명사형 사용 ("취소하기" → "취소") |
| 완료 화면에 경어체 ("완료되었습니다") | 구어체 친근체 ("완료되었어요") |
| 바텀시트 단순 선택형에 UnderlineTab 포함 | 다항목 필터형에만 탭 허용 |
